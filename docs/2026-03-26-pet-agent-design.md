# 微信电子宠物 Agent - 设计文档

> 日期: 2026-03-26
> 状态: 脑爆阶段，持续更新

## 一、项目背景

QQ 宠物（2005-2018）是一代人的青春记忆，每年都有用户呼吁腾讯复活。谷雨曾在腾讯参与 QQ 宠物项目，对产品设计有一手经验。

2026 年腾讯开放了微信 iLink (ClawBot) Bot API，提供了官方合法的个人 Bot 接口。我们计划通过这个接口，在微信聊天窗口中复刻一个电子宠物 Agent。

## 二、核心定位

**情感陪伴型**（A 方案），复刻 QQ 宠物的经典体验：
- 养成系统：有状态值（饥饿、清洁、心情、健康等）
- 日常陪伴：宠物有生命感，需要照顾
- 牵挂感：忘了照顾会有后果

**明确不做（当前阶段）：**
- 社交功能：iLink 接口能力有限，暂无法实现宠物间互动
- 实用助手功能：不做提醒、查信息等工具型功能

**未来方向（记录不实现）：**
- 旅行青蛙模式：宠物从"外面"带回治愈内容（图片、故事、小知识）
- 社交互动：等 iLink 能力扩展后再考虑

## 三、视觉表达方案

### 3.1 核心挑战

微信聊天窗口是"消息流"，发出去不能修改。如何在这个限制下让宠物有"生命感"和"动态感"？

### 3.2 混合方案（确认采用）

**日常互动**：颜文字/Kaomoji + 叙事文字
```
小雪趴在角落，无精打采地舔着爪子 (´-ω-`)
肚子咕咕叫了... 好像很饿的样子
```

**查看状态时**：Emoji 进度条仪表盘
```
🐧 小雪 Lv.3
❤️ 心情：😆 ████████░░ 80%
🍖 饱腹：😐 █████░░░░░ 50%
🛁 清洁：😣 ██░░░░░░░░ 20% ← 该洗澡啦！
⚡ 体力：😊 ██████░░░░ 60%
```

**动作场景**：像素 GIF 动图（需技术验证）

### 3.3 像素动图技术方案（待验证）

**思路**：预制像素 Sprite Sheet → Python 动态裁帧拼装 GIF → 通过 iLink image_item 发送

**Sprite Sheet 设计**：
- 固定宠物形象（如 32x32 像素小企鹅）
- 每个动作/状态 3-4 帧
- 预制动作组：吃饭、睡觉、玩耍、走路、洗澡、生病、开心等
- 生成一次后缓存，不需要每次实时生成

**技术依赖**：
- Python Pillow 库：裁帧、拼 GIF
- iLink image_item (type=2)：图片发送（CDN + AES-128-ECB 加密上传）

**参考项目**：
- [Star Office UI](https://github.com/ringhyacinth/Star-Office-UI) - 海星团队的像素办公室，OpenClaw Agent 可视化
- 区别：Star Office 是 Web 实时渲染，我们是生成 GIF 静态投递

**⚠️ 技术验证优先级 P0**：
1. iLink 能否正常发送图片（015FRIDGE 未踩过这条路）
2. 微信客户端能否正常渲染 GIF 动图
3. 图片上传流程（CDN 预签名 URL + AES 加密）的完整实现

**降级方案**：若 GIF 不可行，退回纯文字（颜文字 + emoji 进度条）

### 3.4 文字表达注意事项

- iLink 文本消息**不支持 markdown**，纯文本
- 支持 emoji 和颜文字
- 微信字体**不等宽**，ASCII art 对齐可能跑偏（█░ 类 Unicode block 字符需实测）
- 文本消息保守上限 2000 字符

## 四、状态系统设计（经典复刻）

### 4.1 核心属性

| 属性 | 说明 | 衰减规则 |
|------|------|----------|
| 饱腹值 | 需要喂食 | 随时间衰减 |
| 清洁值 | 需要洗澡 | 随时间衰减 |
| 心情值 | 需要逗乐/互动 | 随时间衰减，受其他值影响 |
| 体力值 | 活动消耗 | 活动消耗，睡眠恢复 |
| 健康值 | 综合指标 | 其他值过低时下降 |

### 4.2 成长系统

QQ 宠物经典阶段：蛋 → 婴儿 → 幼年 → 少年 → 青年 → 成年
（具体成长曲线待设计）

### 4.3 状态触发行为

- 饱腹值过低 → 宠物主动说饿
- 清洁值过低 → 宠物描述自己脏
- 心情值过低 → 宠物变得沉默/无精打采
- 健康值过低 → 宠物生病，需要治疗
- 全部满值 → 宠物特别开心，表现活跃

## 五、技术架构（参考 015FRIDGE）

### 5.1 三层架构
```
ilink.py  — 通信层：收发消息、维护 cursor、缓存 context_token
core.py   — 核心层：宠物状态引擎、交互路由、持久化
ai.py     — AI 层：性格对话生成、复杂交互理解
```

### 5.2 关键技术选型
- Python + Pillow（图片生成）
- APScheduler（状态衰减定时任务、宠物主动行为）
- JSON 持久化（原子写入：tmp → os.replace）
- AI 大模型（宠物性格/对话，OpenAI 兼容 API）

### 5.3 iLink 关键约束
- context_token 24h 限制：缓存所有 token，发送失败攒到下次用户消息
- 长轮询 35s 超时：APScheduler 独立线程，不阻塞 polling
- 代理绕过：iLink 和 AI API 用 ProxyHandler({}) 绕过 Clash
- 时区处理：zoneinfo.ZoneInfo，不用系统时间

## 七、风险与待定项

| 项目 | 风险等级 | 说明 |
|------|----------|------|
| GIF 渲染支持 | P0 | iLink 图片通道 + GIF 渲染需实测 |

| Token 过期 | P2 | 无刷新机制，需要重新扫码 |
| 腾讯随时可关 | P2 | 条款 7.2，需有降级意识 |
